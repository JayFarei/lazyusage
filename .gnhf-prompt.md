# Objective: Implement the Always-On Collector Daemon (TDD)

Implement the daemon collector and braille graph feature as specified in `kb/plans/02-daemon_collector.md`. Use `/tdd` (red-green-refactor) for every unit of work.

## Approach

Work through the 5 implementation phases in order. For EACH unit of work within a phase:

1. **Red:** Write a failing test first that specifies the expected behavior
2. **Green:** Write the minimum code to make the test pass
3. **Refactor:** Clean up without changing behavior, ensure all tests still pass

Use the `/tdd` slash command to drive each red-green-refactor cycle.

## Validation Commands

- Core unit tests: `bun test tests/core/`
- TUI component tests: `bun run test:tui`
- Type checking: `bunx tsc -p packages/core/tsconfig.json --noEmit`
- All tests must pass before completing each iteration

## Phase 1: Daemon Core (`packages/core/src/daemon/`)

Build the daemon infrastructure in `packages/core/`. Tests go in `tests/core/daemon/`.

### 1a. Schema: `daemon_status` table
- Add `daemon_status` table creation to `packages/core/src/storage/database.ts`
- Test: database creates the table, heartbeat writes/reads work, freshness checks work
- Test file: `tests/core/daemon/status.test.ts`

### 1b. Config loader
- Create `packages/core/src/daemon/config.ts`
- Load from `~/.config/lazyusage/daemon.toml` (optional), merge with CLI flags, apply defaults
- Defaults: interval=60, services=["claude","codex"], log_level="info", pty_recycle_hours=4
- Test file: `tests/core/daemon/config.test.ts`

### 1c. File logger with rotation
- Create `packages/core/src/daemon/logger.ts`
- Write to `~/.local/share/lazyusage/daemon.log`, rotate at 5MB, keep 3
- Test file: `tests/core/daemon/logger.test.ts`

### 1d. Collection loop
- Create `packages/core/src/daemon/collector.ts`
- 60s timer, calls chain.refresh() for each service, writes to UsageStore, writes heartbeat
- PTY warm standby with periodic recycling (configurable, default 4h)
- Resilience: top-level try/catch, never crash on recoverable errors, log and continue
- Test with mocked chain + store: `tests/core/daemon/collector.test.ts`

### 1e. Process lifecycle
- Create `packages/core/src/daemon/lifecycle.ts`
- Fork into background, PID file at `~/.local/share/lazyusage/daemon.pid`
- Signal handling: SIGTERM/SIGINT -> graceful shutdown (winddown PTY, flush DB, remove PID)
- Test file: `tests/core/daemon/lifecycle.test.ts`

## Phase 2: CLI Commands (`packages/cli/src/commands/daemon.ts`)

### 2a. `daemon start` command
- Register `daemon` command group in `packages/cli/src/index.ts`

- Test: `tests/cli/daemon.test.ts` (command registration, flag parsing)

### 2b. `daemon stop` command

### 2c. `daemon status` command
- Check PID file + heartbeat table, format output

### 2d. `daemon logs` command
- Tail log file with --lines and --follow options

### 2e. `daemon install` / `daemon uninstall`
- Generate launchd plist (macOS) or systemd unit (Linux)
- Load/unload the service definition
- Test: verify generated plist/unit file content

## Phase 3: TUI Integration (`packages/cli/src/tui/`)

### 3a. Daemon detection
- In `App.tsx`, check daemon heartbeat on startup
- If daemon is healthy (last_collected_at < 2min), skip own chain initialization
- Read metrics from UsageStore instead
- Test: `tests/tui/hooks/useDaemonDetection.test.ts`

### 3b. On-demand refresh
- `r` key still does a one-shot fetch via chain even when daemon is active
- Test: verify one-shot fetch triggers when daemon is detected

### 3c. Graph tab visibility
- Add "Graph" as 4th tab in StatsPanel, only visible when daemon data exists
- Tab cycles: Daily -> Weekly -> Monthly -> Graph (when available)
- Test: `tests/tui/components/GraphTab.test.ts`

## Phase 4: Braille Chart (`packages/core/src/chart/` + `packages/cli/src/tui/components/`)

### 4a. Braille renderer
- Create `packages/core/src/chart/braille.ts`
- Coordinate-to-braille mapping via bit manipulation (U+2800-U+28FF)
- Canvas abstraction: width x height in cells, plot points, draw lines
- Test: `tests/core/chart/braille.test.ts` (known coordinate -> known braille codepoint)

### 4b. Time-series plotter
- Create `packages/core/src/chart/timeseries.ts`
- Map data points (timestamp, value) to canvas coordinates
- Line drawing between consecutive points (Bresenham or similar)
- Test: `tests/core/chart/timeseries.test.ts`

### 4c. Axes and gridlines
- Create `packages/core/src/chart/axes.ts`
- Y-axis: 0% to scaled ceiling (>100% when prediction exceeds cap)
- X-axis: day gridlines (weekly) or hour gridlines (session)
- 100% threshold as highlighted horizontal line
- Test: `tests/core/chart/axes.test.ts`

### 4d. Chart composition
- Create `packages/core/src/chart/compose.ts`
- Compose: pace diagonal + consumption curve + prediction line + now bar + axes
- Extend existing `packages/core/src/storage/pace.ts` for chart data sourcing
- Test: `tests/core/chart/compose.test.ts`

### 4e. GraphPanel TUI component
- Create `packages/cli/src/tui/components/GraphPanel.tsx`
- Renders composed chart string in a bordered box
- Adaptive: selected metric in panel mode, both weekly+session stacked in fullscreen
- Fullscreen view via `f` key from Graph tab
- Test: `tests/tui/components/GraphPanel.test.ts`

## Phase 5: Integration Testing

### 5a. Daemon lifecycle e2e
- Start daemon, verify PID file + heartbeat, stop daemon, verify cleanup
- Test: `tests/core/daemon/lifecycle-e2e.test.ts`

### 5b. Daemon detection e2e
- Simulate daemon heartbeat in DB, verify TUI reads from DB instead of chain

### 5c. Chart rendering e2e
- Seed database with known snapshot data, render chart, verify braille output matches expected

## Critical Rules

- ALWAYS use `/tdd` for each unit of work
- NEVER skip the failing test step
- Run `bun test tests/core/` after every change to core
- Run `bun run test:tui` after every change to tui
- Run `bunx tsc -p packages/core/tsconfig.json --noEmit` to catch type errors
- Keep iterations small: one test file + one implementation file per iteration
- Export new modules from `packages/core/src/index.ts` as you create them
- Follow existing code patterns (look at neighboring files for conventions)
- The daemon reuses existing infrastructure: PersistentFallbackChain, UsageStore, DedupTracker, PersistentDirectSession
